---
title: Windows
description: Window operations are typed native-service calls with support metadata.
---

Window operations return Effects. They do not throw platform errors into app code.

```ts
const open = Effect.gen(function* () {
  const windows = yield* Window
  return yield* windows.create({ title: "Notes" })
})
```

Renderer code should use the imported RPC group through the app adapter:

```tsx
const window = DesktopApp.useDesktop(WindowRpcs)
const createWindow = window.create.useMutation()
```

The React built-in helpers follow the same supported surface: `windows.create.useMutation()`, `windows.close.useMutation()`, and `currentWindow.close.useMutation()`.

`WindowMethodNames` describes the current Window contract. Some host methods are still unsupported. Those methods remain in `WindowRpcs` with support metadata so docs, tests, and adapters can describe the whole surface while renderer UI can hide unavailable actions.

`WindowSupportedRpcs` is the callable generated group. It currently contains only `Window.create` and `Window.close`.

`WindowBridgeClientOptions` omits `nextRequestId` because generated Effect RPC owns request identifiers for the Window bridge client path.

The generated Window client validates caller input before transport and validates host success payloads before returning app values. Malformed success payloads fail as `HostProtocolInvalidOutputError`.
