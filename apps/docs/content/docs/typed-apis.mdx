---
title: Typed RPCs
description: Native capabilities are modeled as typed services instead of ambient renderer globals.
---

Native work fails for concrete reasons: permission denial, missing files, process exit, invalid paths, unavailable OS services, or closed resources.

Effect Desktop exposes those failures as typed values so callers can handle them deliberately.

```ts
const program = Effect.gen(function* () {
  const filesystem = yield* Filesystem
  const process = yield* Process

  yield* filesystem.writeText({
    path: "/tmp/report.txt",
    contents: "ready"
  })

  return yield* process.spawn("git", ["status"])
})
```

The shape is explicit: services are dependencies, native calls are effects, and failures are part of the type.

## One contract

Renderer-callable APIs start as Effect `RpcGroup` values. `Desktop.Rpc.surface(...)` packages that group into the layers and metadata the framework needs: server layer, generated client layer, deterministic test client layer, schema docs, and executable contract checks.

```ts
export const NotesSurface = Desktop.Rpc.surface("Notes", NotesRpcs, {
  service: NotesClient,
  handlers: NotesRpcs.toLayer({
    "Notes.list": () => Effect.succeed([])
  })
})
```

The group remains the source of truth. Bridge adapters can translate at the host edge, but they should not become a second public contract.

Renderer code should consume RPC groups through React desktop hooks instead of calling `Effect.runPromiseExit` in components:

```tsx
const window = DesktopApp.useDesktop(WindowRpcs)
const createWindow = window.create.useMutation()

createWindow.run({ title: "Notes" })
```

Generated app SDKs should use the same shape for domain RPCs: `notes.createNote.useMutation()`, not `notes.CreateNote.useMutation()`.

Native endpoints also carry support metadata. Use `endpoint.isSupported` before presenting a renderer action when the host implementation may not support the method yet.
