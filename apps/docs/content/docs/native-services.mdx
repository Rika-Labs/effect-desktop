---
title: Native services
description: Native capabilities are Effect services with renderer-callable RPC groups.
---

Native services keep host authority out of renderer code. Runtime programs use Effect services such as `Screen` and `Window`; renderers use imported `*Rpcs` groups through the framework adapter.

## Screen

`Screen` is the current generated native proof. `ScreenRpcs` is the canonical Effect `RpcGroup`, and `ScreenSurface` derives the server, client, test-client, schema-doc, and law artifacts from that group.

```ts
const program = Effect.gen(function* () {
  const screen = yield* Screen
  return yield* screen.getPointerPoint()
})
```

Renderer code derives hooks from the same group:

```tsx
const screen = DesktopApp.useDesktop(ScreenRpcs)
const pointer = screen.getPointerPoint.useMutation()
```

## Window

`Window` exposes typed effects for host window operations. The Phase 5 method contract is visible through `WindowRpcs`, and unsupported methods stay visible as metadata instead of disappearing from docs and descriptors.

`WindowSupportedRpcs` is the callable generated group. It currently contains `Window.create` and `Window.close`; planned chrome mutation, focus, fullscreen, scale, and persistence methods stay descriptor-only until the host implements them.

The generated Window client validates caller input before transport and validates host success payloads before returning app values.

Check endpoint support before presenting renderer actions, and handle runtime failures as typed values.
